TrustSquare — Launch-Readiness Plan

Owner / designer view · 2 June 2026 · scoped to the Pretoria (Tshwane) public launch · managed by the Orchestrator loop

The brief. Everything below is what I'd do next if I owned and designed this — and it deliberately sets aside the external blockers (attorney/EULA, CIPC, FSCA, POPIA, the human-only secrets rotation, the Overpass re-import). Those are parked in their own lane at the bottom so nothing is lost, but they don't gate this plan.

The shape. A marketplace launches when three things are true: it functions end-to-end, it has enough supply, and the first impression is credible. The plan is ordered against exactly those, then handed to the Orchestrator in its own vocabulary so the boring half runs itself and only real decisions reach you.

How the Orchestrator manages this plan

The live loop (Sensor 03:30 → Fixer 04:00 → Orchestrator 05:00, one pass/day) already triages every item into FIX / FILE / IGNORE against two approval gates. I've tagged each action with the lane it lands in, so the plan plugs straight into the existing policy rather than inventing a new process:

AUTO-SHIP verified, non-gated → nightly Fixer deploys it unattended STAGE verified but hits Gate 1 (regulatory) or Gate 2 (financial) → waits for your approve <id> ATTENDED design / feature / structural → FILE'd; built in an attended session (loop never touches it) CONTENT/OPS supply, photos, onboarding, config SET ASIDE external / legal / human-only blocker — parked, not scheduled
The loop only auto-executes the AUTO-SHIP lane. Design, features and structural refactors are explicitly off-limits to it (Policy §7) — so for those, "managed by the Orchestrator" means it tracks and surfaces them on the daily report/dashboard while they're worked in attended waves. The two gates remain the only hard stops, and uncertainty defaults to STAGE.
