Structured Facet Fields — build spec

The proper fix for the Collectors / Cars / Adventures filters · 3 June 2026 · Collectors-first, Scryfall-powered

The problem (root cause of all three filter bugs you found). The per-category facet filters were built in the UI, but for live listings the values exist only as free text in the title. The BEA has no collectible_type, condition, or era column (same for Cars make/year/transmission and Adventures type/duration/group). So the filters have nothing to match on, and the lenient logic shows every listing under every selection.

The fix. Make these real, stored fields — captured once, filtered reliably — instead of guessing from text.

The lever — Scryfall is already wired. The BEA already resolves a Scryfall card id at listing creation (resolve_scryfall_id()) and stores scryfall_id. Since your Collectors inventory is heavily MTG cards, that means type and set-year can be auto-filled from Scryfall — the seller types nothing, and existing cards can be backfilled from their stored id. Only physical condition needs a human (Scryfall can't know the state of the actual copy).

Data model — new listings columns

auto = derived from Scryfall / existing data · seller = chosen at onboarding. All nullable, added via the existing idempotent ALTER TABLE … ADD COLUMN pattern.

Build order — Collectors first (it's what you're testing + has the Scryfall win)

1
BEA — store + serve the fields. Add the Collectors columns to the migration list; accept them on create/edit (POST/PUT /listings); return them in GET /listings. Extend resolve_scryfall_id() to also capture the card's set-year and type at creation. Low-risk: nullable columns, the established pattern. One BEA deploy + restart.
2
FEA — map + filter. Map the new columns onto the browse listing object; bucket era_year → era; add Near Mint to the condition options. The Collectors filters now actually filter. ms.js only — you can test Collectors working at this point.
3
Onboarding — capture them. Add Collectible Type / Condition / Year to the create + edit forms. For cards, type + year arrive pre-filled from Scryfall (seller just confirms + sets condition). Admin/onboarding form change.
4
Backfill — fix the existing listings. The ~26 live Collectors listings: cards get type + year from their stored scryfall_id (a one-time script); condition parsed from the title where present, else flagged for a quick seller/admin pass. One-time data script.

Then repeat Phases 1–4 for Cars and Adventures (same pattern, no Scryfall — all seller-entered).

Holds the scale-shape invariants: nullable columns, standard SQL through the one DB layer, city_id scoping untouched, reads tolerate staleness. Nothing here is a rewrite — it's the data the filters were always meant to read.
