Step 1 Build Spec — The Honest Foundation

TrustSquare Global Marketplace · the one piece needed before launch · 2 June 2026 · v1

What this builds: the listings query that filters, sorts and paginates entirely on the server, scoped by city, returning an anonymity-safe, currency-aware result with stable pagination — and the single canonical "filter spec" that Search (now) and Wishlist (later) will both reuse.

Why it's first: every later piece — the geo window, Wishlist alerts, Tuppence-enhance — sits on top of honest, consistent, global-ready results. Get this right and adding a country (or a million listings) is a config change, not a rewrite. Scope is deliberately tight — no ranking, no window composition, no alerts here.

1Problem

Three foundation cracks, all in the buyer feed (GET /listings in bea_main.py):

  • Filters lie at volume. The server returns up to a fixed slice; the browser (renderGrid) then filters by price, trust, beds, etc. on whatever it received. So in a busy city a filter like "drill ≤ R5k, trust ≥ 70" only searches the loaded slice — real matches get hidden. A backend limit is changing what a filter means, which is wrong.
  • No currency/country on a listing. Prices are free-text strings; there is no per-listing currency or normalised price, so cross-country comparison and an honest price-band filter are impossible. This blocks "global" entirely.
  • The buyer response leaks the seller. The endpoint returns every row column verbatim — including seller_email — to buyers. That violates the absolute anonymity rule (A2): seller identity must never reach a buyer.

Step 1 fixes all three so the feed is consistent, global-ready, and anonymous — the bedrock for Steps 2–4.

2Goals (measurable)

  • A filter returns the complete, correct match set for a city regardless of how many listings exist (10 or 10,000) — proven where matches exceed one page.
  • One canonical filter spec + matcher — the exact query Search uses and Wishlist will reuse, with zero duplicate matching logic.
  • Every listing carries a currency + resolvable country; the price band filters in the buyer's currency.
  • Zero seller-identifying fields in any buyer-facing response.
  • Pagination is stable (no duplicates or skips) and scales (keyset, not offset).
  • Adding a country is config — no query rewrite (scale-shape invariant #5).

4User stories

  • Buyer: "I want my price/trust/category filter to return every match in my city, so I don't miss options as the city grows."
  • Buyer: "I want to scroll a long result list smoothly — no duplicates, no gaps — even as new listings appear."
  • Buyer (global-ready): "I want prices shown in my own currency so I can compare like-for-like."
  • Seller: "I want my email and identity never exposed in buyer search results."
  • Platform: "I want one canonical listings query, so Search and Wishlist match listings with identical logic and can't drift apart."

5Requirements

P0 Must-have — the feature cannot ship without these

IDRequirementAcceptance criteria
R1Core filters move server-side — category, price_min/max, trust_min, suburb applied in SQL (not the browser).Given a city with more matches than one page, the total matches the engine reports equals the true DB count for that filter — independent of page size. R3Anonymity-safe response. Return a whitelist of buyer-safe fields only; strip seller_email and any internal/PII columns.The GET /listings JSON contains zero seller_email (and no other seller-identifying field) for any query. Verified by an automated assertion + a one-off audit.
R4Per-listing currency + country. Add a currency (ISO-4217) column, backfilled ZAR; country resolvable via the geo chain (or denormalised country_iso2).Every listing row has a non-null currency; the response includes currency; country derivable for every live listing.
R5Currency-aware price band. price_min/max interpreted in the buyer's display currency; normalisation hook present (pass-through while single-currency).With one currency, behaviour is identical to a plain numeric band. The contract carries a currency so a second currency is a config add, not a code change.
R6Canonical response contract. { listings:[…], page:{ next_cursor, has_more } }; the FEA reads .listings and drops client-side band filtering.FEA renders correctly from the new shape; a compat shim tolerates the old bare-array during cutover; smoke 39/39.
R7Standard SQL, thin layer, city-scoped (scale-shape #1 & #5). Parameterised; no SQLite-only pragmas in the handler; always scoped by city_id.Query is parameterised + standard SQL (SQLite→Postgres is a connection swap); every query path carries city_id.

6Technical contract

Endpoint — evolve GET /listings

ParamMeaning (Step 1)
city / city_idThe buyer's scope anchor (required). Step 1 = single city; scope param reserved for Step 2 (city·region·country·global).
categoryOptional category filter (already server-side today).
price_min, price_maxNumeric band, in currency below.
currencyBuyer's display currency (default = city's currency). Drives band normalisation + display.
trust_minMinimum trust score (e.g. 70).
suburbOptional suburb narrow.
demoExisting seed-data toggle (unchanged).
cursorOpaque keyset cursor (encodes last created_at,id + a filter signature). Absent = first page.
page_sizeDefault 24, max 100.

Response shape

// buyer-safe whitelist — no seller_email, no internal columns
{
  "listings": [
    { "id", "title", "price", "currency", "price_display",
      "category", "city", "suburb", "area", "trust_score",
      "thumb_url", "photos", "beds", "baths", "listing_lat", "listing_lng",
      "nearby_pois", "linked_wonders", "created_at" }
  ],
  "page": { "next_cursor": "…", "has_more": true }   // total: P1
}

Sort + keyset (the stable-pagination core)

ORDER BY created_at DESC, id DESC
-- next page: carry the last row's (created_at, id)
WHERE (created_at, id) < (:last_created, :last_id)   -- tuple compare
LIMIT :page_size

The sort key is deliberately a tuple ending in id so it's unique + stable, and so Step 2 can prepend a relevance score (ORDER BY score DESC, created_at DESC, id DESC) without changing the pagination mechanism.

Data model

  • ALTER TABLE listings ADD COLUMN currency TEXT DEFAULT 'ZAR' — backfill existing rows to ZAR. (Optionally also country_iso2 denormalised for query speed — see open question.)
  • Index for the keyset path: (city, listing_status, created_at, id) (+ category if filtered often).
  • The query stays inside the existing thin DB access path — no SQLite-only pragmas leak into the handler.
Migration / rollout (important — touches the core feed + big files): the BEA change (bea_main.py) and the FEA change (ms.js loadLiveListings + drop client-side band filtering) must land together, since the response shape changes. Use a compat shim (accept bare-array or wrapped) during cutover. The offset pagination already deployed is the interim — Step 1 upgrades it to keyset + the contract. All edits via the Python str-replace driver (never Edit/Write on the big files), ast/node --check + smoke before deploy, and applied to the correct base given the pending-TVS divergence. Attended build, not auto-ship.
