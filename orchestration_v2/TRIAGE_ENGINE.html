Triage Engine — Phase 2

Turns a raw Detect run into a governed, de-duplicated, prioritized queue. It decides; it never fixes.

🟢 GREEN LANE · deterministic · zero-token · the connective tissue Detect → Fix

What it is

Detect finds; Triage governs. Every raw finding is de-duplicated against what we already know (the backlog, the dismissed-list, the changelog), given a lane (who may act) and a priority (how urgent), and laid out on a board. Known issues are never re-reported; dismissed false-positives never come back to bother you; only genuinely new, genuinely actionable work moves forward — and the green lane flows straight into Fix.

How you call it

> Run the Triage engine on the latest Detect findingsdeterministic helper: triage.py --findings <run>.json → triaged.json + triage_board.html

How it runs

1
Dedupe — against the second brainEach finding gets a stable key = file ∷ root-token (line numbers excluded — they drift). Matched three ways: the backlog (already filed → reuse the id, never duplicate), the ignore-list (already dismissed → drop, reason kept), and the changelog (already fixed → resolved; re-detected later → a regression).
2
Classify a lane — red checked firstThe policy's path pre-catch: anything touching money, secrets, permissions, deletions, public actions or schema is red before intent is even read. Green only for confirmed-safe classes; everything else amber. Fail-safe: ambiguous never reaches green.
3
Prioritize — P1 · P2 · P3Urgency, orthogonal to the lane. P1 breakage/crash/security, P2 degraded/latent, P3 dead-code/cosmetic. Orders the work within each lane.
4
Write & surfaceThe governed queue goes to triaged.json (what Fix consumes) and the visual triage board in your cockpit. Known/dismissed items sit out of the way, kept — never re-raised.

What a triaged item looks like

dedupe_key: file ∷ root-token        verdict: new | known-filed | known-ignored | resolved | regression
lane:       green | amber | red       priority: P1 | P2 | P3
ref:        the backlog / ignore id   status:  queued (→Fix) | staged (→you) | dropped

The three lanes — who may act (approved once, in Phase 0)

🟢 Green

auto-ship — reversible, well-understood.

demo data · FE cosmetic · dead code · docs · poka-yokes. → Fix consumes this lane.

🟡 Amber

do + notify — reversible, 1-click undo.

backend that passes tests · verified data change.

🔴 Red

always asks — never pre-authorised.

money · Tuppence ledger · payments · secrets · permissions · deletions · public posts · schema. Never auto-consumed.

Guardrails

🟢
Decides, never fixesTriage changes no code. The green queue it produces is what the Fix stage acts on, under the lane rules.
🔴
Fail-safe to redPositive confidence is required to call something green. Ambiguous, needs-confirm, or any red signal → it never auto-ships. Silence is not a pass; a red proposal is never downgraded.
🔌
Deterministic & off-gridA zero-token Python script — auditable rule tables, no model in the loop. A model is consulted only for the rare item a rule genuinely can't classify.
